# 「跑步听什么」产品设计与需求判断

2026-07-04 基于中/英文市场调研(关键词、竞品、真实用户讨论)+ 现有管线能力写成。
结论先行:**需求真实且被大规模验证,但"独立 AI 播客 App"这条路已被证伪三次;可行定位是「你自己信息源的音频界面」,以私有播客 feed + 场景化播放器交付,极客开源分发起步。**

---

## 1. 一句话定位

> 把**你自己订阅的信息源和书**,每天自动变成能在跑步/通勤时连续收听的私人电台。

不是"AI 播客节目"(内容),是"信息获取的音频界面"(效率工具)。用户对自己选定的源没有"真实感/人格魅力"预期——要的是效率,不是灵魂。

**非目标(明确不做):**
- 不做泛内容 AI 电台(来福:前百川联创、日均下载仅百次,已证伪)
- 不做面向播客创作者的制作工具(Jellypod/Wondercraft 的 $25-47/月红海)
- 不做播客收听平台(ChatPods/小宇宙的地盘)

## 2. 需求判断(调研摘要)

### 2.1 需求为真的证据

| 证据 | 强度 |
|---|---|
| NotebookLM Audio Overviews:HN launch 907 分/490 评论,~45% 用户日常使用对话播客功能 | 品类级验证 |
| ebook2audiobook 12.6k GitHub stars;HN「Kokoro 生成有声书」420 分、「Abogen」349 分 | 整本书→有声书 DIY 需求确凿 |
| ListenHub(红杉投)用户嫌 $9.9/月"不够用"被迫出 $19 档 | 重度收听者用量远超按条计费 |
| 知乎「跑步时听什么书」2015 年问题至今活跃;「公众号文章朗读软件」直接提问 | 场景需求十年级稳定 |
| Readwise Reader 用户反复要"整个稍后读队列连续听"而不得 | 队列连听是未被满足的行为 |
| Spotify 2026-05 上线 Personal Podcasts;WaPo 用 ElevenLabs 做个性化播客 | 巨头确认"个人 AI 音频填充碎片时间"是主流行为 |

### 2.2 需求为真但生意难做的证据(死亡名单)

| 死者 | 死法 | 教训 |
|---|---|---|
| Recast(文章→对话音频,2024-10 关停) | 无融资,功能被大厂商品化 | 纯"文章转音频"消费 App 单点不成立 |
| PlayNote/PlayAI(2025 被 Meta 收购后关停) | API 用户被抛弃 | 依赖它的下游一夜死 |
| Huxe(NotebookLM 原班人马,$4.6M,2026-05-22 关停) | Spotify 上线次日死 | 连原创团队+顶级投资都扛不住平台入场 |

### 2.3 幸存者模式

- **Audioread**(~文章→私有播客 RSS):小而活,口碑是"rock-solid podcast feed"——便宜、粘性、管线型
- **Speechify**:6000 万用户、盈利——单人朗读大众市场已被占
- **ebook2audiobook**:开源自部署,12.6k stars——极客侧靠免费+自托管聚人
- **Podwise**($12k ARR 起步):中文开发者独立产品能跑通付费,但靠海外用户

### 2.4 空白点(两份调研交叉确认)

1. **订阅源→自动定期生成→播客 feed 的闭环无人做。** NotebookLM/豆包/GenFM/Spotify 全是"手动投喂单篇→单集"的 pull 模式;没有 to-C 产品做 push 式"绑定我的源,每天醒来自动有一期"。中文侧开源 PodLM、英文侧 Reddit 上一堆 NotebookLM 自动化 hack,证明用户在自己拼这个闭环。
2. **整本书→有声书低门槛产品缺位。** NotebookLM 截断长文档("书后半本被含糊带过"是头号投诉);微信读书锁自家书库;唯一整本方案是高门槛开源。「自己的 epub 整本预生成+断点续播」正好落在 SaaS 与 DIY 之间的空档。
3. **跑步/通勤场景化设计无人认领。** 所有竞品的"通勤时听"只是文案;没人做锁屏优先、睡眠定时、按跑步时长配集长、免手队列。「跑步听什么」「what to listen to while running」是纯内容意图的高流量搜索词,零工具产品在争。
4. **成本结构是结构性优势。** 竞品按条/按 credit 计费(Wondercraft 免费档 72 分钟/**年**),每日个人收听量直接把用户价格打穿;我们管线级成本近零,是能做"日更不限量"定价的底气。

## 3. 目标用户与场景

**核心画像**:信息重度消费者 × 规律运动/通勤者。订着 RSS/newsletter/公众号、稍后读堆积如山、每周跑 3-5 次或单程通勤 40 分钟+,"不愿浪费碎片时间"。先期是极客(会自部署、能提供 OPML),后期泛化到"给我个网址就行"的普通用户。

**三个场景闭环:**
1. **晨跑/通勤电台**:前一晚管线自动跑,出门时今天的 5 集已在播放器/播客 App 里,戴上耳机直接连播。
2. **啃书**:上传 epub/mobi→整本预生成→跑长距离时连续听,断点自动记忆。
3. **稍后读清空**:攒了一周的文章,一次转成周末长跑歌单(v1 功能)。

## 4. 差异化与护城河

| 维度 | 竞品现状 | 我们 |
|---|---|---|
| 生成触发 | 手动投喂单篇 | **定时管线自动跑**(systemd timer,已上线) |
| 交付 | 锁在自家 App 里 | **标准播客 RSS feed**——Apple Podcasts/小宇宙外的任何客户端都能订 + 自有 Web 播放器 |
| 书 | 截断/锁书库/DIY 高门槛 | **整本预生成 55 集级别**,断点续传生成+续听(已上线) |
| 场景 | 文案提一句"通勤" | 锁屏控制、睡眠定时、倍速、跨集连播、位置记忆(已上线) |
| 成本 | credit 制,日更即破产 | TTS/LLM 成本近零,可承诺"你的源,日更不限量" |
| 内容形态 | 单人机械朗读 或 注水对话 | 对话式(资讯)+ 直读(书)按内容类型分,对话稿限 600-900 字防注水 |

护城河诚实评估:功能本身无壁垒(Google/Spotify 一个季度能抄完),真正的防御是 **(a)** 私有 feed 的迁移粘性 + **(b)** 大厂不屑做的"自带源+自托管"极客定位 + **(c)** 「跑步听什么」这个中文 SEO 心智(产品名=搜索词)。

## 5. 路线图

### v0(现状,已全部上线)
定时管线(6 源轮询、429 重试)→ LLM 对话稿(laozhang.ai)→ Edge TTS 双声 → Supabase feed.xml + books.json → 静态播放器(Cloudflare Pages)(资讯/书籍、睡眠定时、锁屏、断点)。

### v1 —— 从"自用工具"到"可给别人用"(顺序即优先级)
1. **播客 App 订阅打通**:feed 加 iTunes 封面/分类,验证 Apple Podcasts「通过 URL 关注」全流程(现有 feed 已可订,补齐元数据)
2. **单篇即时转**:粘贴 URL→生成一集插入 feed(补齐"稍后读清空"场景;也是所有竞品用户的既有习惯)
3. **书籍对话伴读(可选)**:每本书生成一集 15 分钟"两人聊这本书"导读,直读正片前先听——差异化 NotebookLM 只能做导读、我们导读+全本都有
4. **PWA + 离线缓存**:跑步场景弱网是常态,Service Worker 预缓存下一集
5. **源管理 UI**:目前改 config.json,做成页面上加/删 RSS
6. **多用户隔离**(决定商业化后才做):每用户一个 bucket 前缀 + 私有 feed token

### v2 —— 场景纵深(验证 v1 留存后)
- 按跑步时长点单("今天 40 分钟"→自动凑集)
- 公众号源接入(用户提供合规抓取途径)
- 中文 SEO 内容站:「跑步听什么」词族的推荐内容+工具入口
- 声音可选(用户已明确暂缓,放这里不忘)

### 商业化前置的硬改造(不商用不动)
- **TTS 换商用通道**:Edge TTS 是免费非授权通道,仅限自用;商用换 MiniMax/阿里(接口已抽象成 TtsProvider,换实现即可)
- 存储出 Supabase 免费档(R2/B2,流量成本敏感)
- 资讯源版权评估:全文改写→对话是灰区,商用需摘要+链接原文的形态

## 6. 关键词/长尾词弹药库(SEO/命名用)

**中文(找内容,高流量,内容站争夺)**:跑步听什么 / 跑步 播客 推荐 / 通勤听什么 / 跑步 听书 / 马拉松 训练 听什么
**中文(找工具,精准转化)**:文章转语音 / 公众号 转 语音 / 文字转播客 / AI 播客生成 / RSS 转播客 / epub 转 有声书 / NotebookLM 平替 / NotebookLM 国内
**英文(找工具)**:articles to podcast / newsletter to podcast / RSS to podcast / read it later audio / epub to audiobook / NotebookLM alternative / personalized daily podcast
**英文(找内容,零工具竞争)**:what to listen to while running / best podcasts for long runs

命名巧合:产品名「跑步听什么」本身就是最高流量的中文场景词;域名 listenwhilerunning 同理对应英文场景词。这个心智位要占住。

## 7. 商业化三条路(有真实 trade-off,需拍板)

| 路径 | 玩法 | 优 | 劣 |
|---|---|---|---|
| **A. 开源自托管 + 托管版收费(推荐)** | 管线开源发 Show HN/知乎(对标 ebook2audiobook 的 12.6k stars 分发),不想折腾的用托管版,小额订阅(≤20 元/月 或 $5-8/月,对标讯飞 12 元档与 Audioread) | 获客零成本走开源口碑;极客定位避开大厂正面;托管版收的是"省事钱",平台抄功能抄不走自托管人群 | 开源后功能无秘密;托管版有 TTS 商用成本与版权责任;天花板是 lifestyle business 量级 |
| B. 纯 SaaS 订阅 | 直接做多用户产品收费拉新 | 商业模型直白 | Recast/Huxe 的原路;获客成本无解;Spotify/豆包免费碾压 |
| C. 保持自用 + 内容化 | 工具不商业化,做「跑步听什么」内容号/站,流量变现或导流 | 零改造成本零风险;SEO 词族无竞争 | 是媒体生意不是产品生意;工具价值不变现 |

**推荐 A,且分两步走**:先把 v1 的 1-4 做完(自己是第一个重度用户,产品被自己的跑步磨到顺),再开源+Show HN 试水;有真实外部用户再谈托管版。B 的死亡案例太多,C 可以作为 A 的获客手段而非独立路线。

## 8. 已知风险与设计对策

| 风险 | 对策 |
|---|---|
| 平台风险:Spotify/Google/豆包随时把功能免费化 | 不做"通用生成",做"自带源+自托管+运动场景"的窄定位;私有 feed 粘性 |
| 对话注水(NotebookLM 头号投诉:5 分钟内容拉到 20 分钟) | 对话稿硬限 600-900 字;书籍走直读不硬凑对话 |
| 新闻幻觉(NPR 点名 WaPo AI 播客准确性) | 每集尾部报来源名(已做);商用形态改"摘要+原文链接" |
| Edge TTS:非商用授权 + 节流 | 自用期已有 5 次退避+间歇;商用换 MiniMax(TtsProvider 接口已预留) |
| 版权:全文改写他人内容商用 | 自用无虞;商用前法务评估,形态退到摘要级 |
| 中文付费意愿低(Podwise 靠海外) | 英文市场同步(listenwhilerunning 域名/英文源已支持),定价对标 Audioread 而非 ListenHub |
