# 路线对比与决策记录

2026-07-02 调研后的全部候选路线。已采纳:**路线二 → B → A** 演进线。

## 不做产品的三条路

### 路线一:现成工具直接用(零成本,满足度 60-70%)

| 工具 | 用法 | 短板 |
|---|---|---|
| NotebookLM(已支持中文) | 文章/文档批量生成 Audio Overview,跑步前生成好 | 批量生成 ≠ 连续听;双人对话 AI 味 |
| 豆包 / 腾讯混元 AI 播客 | 链接/文档一键转 2-10 分钟播客 | 单集工具,没有流 |
| Snipd / 小宇宙 | 真人播客 + AI 章节/摘要 | 内容不个性化,但质量天花板高 |
| 微信读书 TTS / 有声书 | 长内容连续听 | 无时效性内容 |

### 路线二:自建私人播客 feed(半天,满足度 90%)★ 已采纳为第一步

cron 抓 RSS → Gemini 口语化改写 → TTS → 私有 RSS feed → 播客 App 订阅。

- 播放器基础功能白嫖(Huxe 死因那一层直接外包)
- 自用非商用 edge-tts 无许可问题;成本约等于零
- RSS 源即兴趣画像,个性化自己配置
- **隐藏价值:就是 MVP 的 dogfooding**。三个月自己都留不住 → 验证了来福/Huxe 死因,零沉没成本;离不开 → 进入 B

### 路线三:不做 App,做内容

AI 辅助播客频道发现有平台。单位经济已验证($1/集、20 听众回本,Inception Point 模式)。避开硬新闻和娱乐。风险:podslop 政策收紧(Apple 要求 AI 披露)、同质化。

## 自研产品的三个形态

### A. 跑步电台 App/PWA(消费级)

场景绑定跑步、无限连播、单人旁白、跳过反馈。

- 依据:全赛道无人做"场景感知连续流";跑步社区策划摩擦是实证痛点(133 条评论:每月手动建歌单、跑中切播客/音乐)
- 代价:直面来福/Huxe 死因,留存硬仗;消费级付费意愿未验证(ListenHub 创始人:消费侧单位经济不成立)
- 形态:PWA 优先(栈复用 + Media Session API 锁屏控制 + 免审核),跑通再原生

### B. "RSS → 私人电台" prosumer micro-SaaS ★ 推荐优先

路线二的产品化:用户绑定 RSS/收藏夹/newsletter,每天生成私人音频 feed,用户自己的播客 App 收听。

- 依据:中文 AI 音频已验证的付费全在工具侧(ListenHub 付费转化率远超国内 C 端均值、Pro 年费 1000+);"内容预生成好"天然满足 Snipd 教训
- **关键优势:不养播放器、不养推荐算法、不打留存硬仗**——只做内容管线,恰好是独立开发者能做好的那一层
- 代价:天花板低(prosumer 小众);豆包/混元功能嵌入持续挤压;差异点=自动化+私有 feed,不是"能生成"
- 变现:订阅 $5-8/月;每用户每天 20-30 分钟音频,MiniMax 计价下毛利健康

### C. 微信生态形态(A 的低摩擦变体)

小程序/公众号推送音频。依据:独立 App 获客是第一道鬼门关(来福 2000 下载);微信自己在灰度"快讯"AI 播客。代价:小程序后台播放/锁屏受限——跑步场景恰恰最需要;活在腾讯屋檐下。

## 排序理由

**B 是唯一"已验证有人付费 + 避开留存硬仗 + 工程能力全覆盖"三者同时成立的路。** A 的空白是真的,但空白里的两具尸体(Huxe、来福)也是真的——等路线二 dogfooding 三个月后用自己的留存数据决策。这也是 ListenHub 的路径(先工具积累技术和数据,电台放后面),区别是不用融资,可以走更小步。

最坏情况:花半天得到一个自己每天在用的工具,零沉没成本。
